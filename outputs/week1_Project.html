Task 1 Written Explanation

What a single row represents: An invoice line item found in real-world business operations. The document presents a particular product item that was purchased but it lacks information about the entire transaction. The database will create five distinct entries for a customer who purchases five different items during a single shopping visit.

Invoice ID: Multiple rows share the same Invoice number because they belong to the same "basket" or transaction. It shows that those different products were all bought at the same time by the same person.

Customer ID: Multiple rows share the same Customer ID because a customer may have multiple items in one basket, and they may also return to the store to make entirely new purchases on different dates.

Task 2 Define the Unit of Analysis

In this task, I evaluate three different ways to structure the data for analysis. The appropriate unit of analysis must be selected because it defines the patterns that machine learning models will detect.

  1. Invoice Line (Row Level): The dataset currently displays its complete contents because each invoice field contains one product. The system preserves complete product information through StockCode and unit price details which enable users to track their shopping patterns. The complete understanding of a business transaction becomes unreachable because of this information loss. The system requires manual row grouping to determine the total shopping value and to track customer behavior patterns over time.

  2. Invoice / Basket (Aggregated by Invoice): The data analysis process achieves its objective because each shopping trip gets represented through the Invoice-based data grouping method. The system provides two types of data which include "Basket Size" that measures items per trip and "Total Invoice Value" which shows overall purchases. The system provides excellent performance assessment tools which enable users to evaluate store operations throughout various time periods which include daily and hourly schedules. The specific product details become inaccessible through this method. The basket cost of 100 becomes known to us yet we lose the ability to determine whether the 100 value originated from one expensive item or from fifty inexpensive items.

  3. Customer (Aggregated across all Invoices) The data shows customer purchases through their unique Customer ID which creates a complete shopping record for each customer. Through this method we can measure Customer Lifetime Value (CLV) and track customer purchasing patterns and their degree of loyalty. The method provides the ideal standard for "Customer Segmentation" which enables companies to cluster their customer base according to their worth. The process results in complete elimination of all time-based information. The system now restricts us from tracking customer shopping habits which include their seasonal behavior patterns and their preference for weekend shopping over weekday shopping.

Decision My chosen unit of analysis: Customer Level. Justification: I have selected Customer as my primary unit of analysis for all upcoming project work. Retail businesses achieve their most useful insights through customer base analysis instead of studying single customer purchases. By aggregating to the customer level, I can prepare for future tasks like identifying high-value customers, predicting churn, or creating targeted marketing segments based on their total spending and visit frequency.

# --- TASK 2 CODE: Implementing the chosen Unit of Analysis (Customer Level) ---

# First, we create a 'TotalSpend' column (Quantity * Price) 
# so we can see the total money spent per row.
df['TotalAmount'] = df['Quantity'] * df['Price']

# Now we aggregate the data to the Customer Level.
# We want to know: How many trips (Invoices) did they make? 
# How many items (Quantity) did they buy? 
# And what is their total lifetime spend (TotalAmount)?

customer_level_df = df.groupby('Customer ID').agg({
    'Invoice': 'nunique',     # Count unique invoice numbers (Frequency of visits)
    'Quantity': 'sum',        # Total number of products bought
    'TotalAmount': 'sum'      # Total revenue generated by this customer
}).reset_index()

# Rename columns to be more descriptive for the new unit of analysis
customer_level_df.columns = ['CustomerID', 'TotalVisits', 'TotalItemsBought', 'LifetimeSpend']

print("--- Data aggregated to the CUSTOMER level ---")
print(f"Original rows: {len(df)}")
print(f"New rows (one per customer): {len(customer_level_df)}")
print("\nFirst few rows of the new Customer-Level dataframe:")
display(customer_level_df.head())

Task 3 Data Quality and Validity Audit

Three primary data quality problems relevant to my study's validity have surfaced as I interact with the Online Retail dataset.

Issue 1: Missing Customer Identifiers The issue exists because 107927 rows in the dataset contain null Customer ID values. The Customer serves as my primary research unit because research requires customer identification to connect user information with their loyalty records. The data contains "anonymous" information which does not show how individual customers behave because it includes these specific rows. The solution requires me to filter out these rows through removal. Without a Customer ID these observations lose their value for a customer-centric model. The removal of these elements guarantees that all records in my final analysis link to a customer journey which can be tracked.

Issue 2: It involves handling negative quantities which occur during returns and cancellations. The system contains 12326 records which show negative values in their Quantity column. The returned items and canceled orders usually lead to these negative quantity entries. The negative quantities cause distortion in the "Total Items Bought" and "Lifetime Spend" metrics. My audit uncovered 90 customers who show negative total spend which should not exist in a sales environment. I will eliminate these negative transactions from my analysis to concentrate on analyzing successful sales.

Issue 3: There are 3,687 rows which record the Price value as 0.0. The transactions with zero prices do not generate any revenue which explains their significance. The records When combined with actual sales The records will reduce the Average Price and Average Order Value metrics. I will filter (remove) rows where the price is less than or equal to zero. The transactions do not qualify as business operations. The elimination of these records will enable my revenue metrics to show actual business sales.

# Fix the Date column - added dayfirst=True to fix the ValueError

# 1. TEMPORAL PATTERN
